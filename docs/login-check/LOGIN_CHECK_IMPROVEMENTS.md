# 登录状态检测功能改进

## 新增功能

### 1. 批量选择检测

在"账号登录状态"列表中：
- **复选框**：每个账号前都有复选框，可以选择要检测的账号
- **全选/取消全选**：点击表头的复选框可以全选或取消全选
- **批量操作栏**：选中账号后会显示批量操作栏

### 2. 灵活的检测模式

#### 检测所有账号
- 点击"立即检测"按钮
- 可选择是否发送飞书通知
- 默认串行检测（一个接一个）

#### 检测选中账号
- 勾选要检测的账号
- 在批量操作栏中配置：
  - **发送飞书**：是否发送飞书通知
  - **并行检测**：是否同时检测多个账号
- 点击"检测选中账号"按钮

#### 检测单个账号
- 点击账号行的"检测"按钮
- 立即检测该账号（不发送通知）

### 3. 串行 vs 并行检测

#### 串行检测（推荐）
- ✅ 一次只打开一个浏览器窗口
- ✅ 检测完成后自动关闭，再检测下一个
- ✅ 资源占用少，稳定性高
- ⚠️ 检测时间较长（账号数 × 单个检测时间）

**适用场景：**
- 账号数量较多（5个以上）
- 服务器资源有限
- 不着急看结果

#### 并行检测
- ✅ 同时打开多个浏览器窗口
- ✅ 检测速度快
- ⚠️ 资源占用大
- ⚠️ 多个窗口需要通过标题区分

**适用场景：**
- 只检测少数账号（2-3个）
- 需要快速得到结果
- 服务器资源充足

### 4. 浏览器窗口区分

每个浏览器窗口的标题会显示账号信息：
```
登录检测 - account_xxx
```

无论是串行还是并行检测，都能清楚地知道每个窗口对应哪个账号。

## 检测流程

### 已登录状态
1. 打开浏览器
2. 访问微信视频号平台
3. 检测到已登录（URL 或 UI 匹配）
4. 等待 2 秒
5. **自动关闭浏览器** ✓

### 需要登录状态
1. 打开浏览器
2. 访问微信视频号平台
3. 检测到需要登录
4. 捕获二维码图片
5. 等待 180 秒（3分钟）
6. 超时后**自动关闭浏览器** ✓
7. 如果勾选"发送飞书"，会发送通知（包含二维码图片）

## 使用场景

### 场景 1：日常巡检（检测所有账号）
1. 进入系统设置页面
2. 不勾选"发送飞书"（避免打扰）
3. 点击"立即检测"
4. 系统串行检测所有账号
5. 查看检测结果

### 场景 2：快速检查个别账号
1. 在账号列表中勾选要检查的账号（如 2-3 个）
2. 勾选"并行检测"（快速）
3. 不勾选"发送飞书"（测试用）
4. 点击"检测选中账号"
5. 多个浏览器同时打开，通过标题区分

### 场景 3：紧急处理掉登录
1. 发现某个账号可能掉登录
2. 点击该账号行的"检测"按钮
3. 立即检测该账号
4. 如果需要登录，手动扫码

### 场景 4：批量检测并通知
1. 勾选需要检测的账号
2. 勾选"发送飞书"
3. 选择"串行检测"或"并行检测"
4. 点击"检测选中账号"
5. 如果有账号掉登录，会发送飞书通知（包含二维码）

### 场景 5：定时自动检测
- 系统会按照配置的间隔（默认 30 分钟）自动检测所有账号
- 自动检测采用**串行模式**
- 自动检测**始终会发送飞书通知**（如果状态变化）
- 这是为了确保掉登录时能及时通知到相关人员

## 功能对比

| 功能 | 立即检测 | 检测选中账号 | 单个检测 |
|------|---------|-------------|---------|
| 检测范围 | 所有账号 | 选中的账号 | 单个账号 |
| 检测模式 | 串行 | 串行/并行可选 | 单个 |
| 飞书通知 | 可选 | 可选 | 不发送 |
| 使用场景 | 全面巡检 | 批量检查 | 快速检查 |
| 浏览器窗口 | 依次打开 | 串行依次/并行同时 | 单个窗口 |

  "parallel": true

```http
POST /api/login-status/check/:accountId
```

### 1. 修复登录状态判断逻辑
**之前的问题：**
```python
# 判断 dashboard 需要 URL 和 UI 都匹配（太严格）
if dashboard_url and dashboard_ui:
    has_dashboard = True
```

**修复后：**
```python
# 判断 dashboard 只需 URL 或 UI 任一匹配即可
if dashboard_url or dashboard_ui:
    has_dashboard = True
```

这样即使 UI 选择器没匹配到，只要 URL 正确也能识别为已登录状态。

### 2. 优化检测优先级
```python
# 优先判断 dashboard（已登录状态）
if dashboard_url or dashboard_ui:
    has_dashboard = True
elif login_url or login_ui:
    has_login = True
```

已登录状态的优先级高于需要登录状态。

### 3. 浏览器标题设置
```python
page.evaluate(f"""() => {{
    document.title = '登录检测 - {account_display}';
}}""")
```

在页面加载后立即设置标题，方便用户区分多个窗口。

## 配置说明

### 环境变量
```bash
# 登录检测开关
LOGIN_CHECK_ENABLED=true              # 是否启用定时检测
LOGIN_CHECK_INTERVAL_MINUTES=30       # 检测间隔（分钟）
LOGIN_CHECK_RETRY_TIMES=3             # 失败重试次数

# 飞书通知开关
FEISHU_NOTIFY_LOGIN_STATUS=true       # 定时检测时是否发送通知
```

### 前端控制
- 手动检测时的"发送飞书"复选框**仅影响本次检测**
- 不会改变定时检测的通知设置
- 定时检测的通知由 `FEISHU_NOTIFY_LOGIN_STATUS` 环境变量控制

## 注意事项

1. **串行检测**：多个账号会依次检测，不会同时弹出多个浏览器
2. **自动关闭**：浏览器会在检测完成后自动关闭（已登录 2 秒，需登录 180 秒）
3. **窗口标题**：每个浏览器窗口标题显示账号信息，方便区分
4. **通知控制**：手动测试时可以选择是否发送飞书通知
5. **定时检测**：定时检测始终会发送通知（如果状态变化），确保及时告警

## 故障排查

### 问题：浏览器不自动关闭
- 检查是否修复了登录状态判断逻辑（dashboard_url **or** dashboard_ui）
- 查看控制台日志，确认状态判断结果
- 检查 UI 选择器是否匹配当前页面

### 问题：无法区分多个浏览器窗口
- 确认 Python 脚本已更新（包含 page.evaluate 设置标题的代码）
- 检查浏览器标题栏是否显示"登录检测 - xxx"

### 问题：手动检测时仍然发送通知
- 确认前端代码已更新（包含 testNotifyFeishu 变量）
- 确认后端代码已更新（支持 notifyFeishu 参数）
- 检查"发送飞书"复选框是否未勾选
